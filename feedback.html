What's Working Well — Strengths

🤖 Multi-AI Backend

  • Gemini (best quality, rate-limit retry with exponential backoff)
  • G4F (free, 13+ models including DeepSeek, Claude, Llama)
  • Local GGUF via llama.cpp — true offline usage
  • Manual copy/paste fallback for any LLM

👤 Smart Face Detection

  • Dual engine: InsightFace (primary) + MediaPipe (fallback)
  • 1-face and 2-face split-screen modes with auto-detect
  • Face identity lock prevents position swap in 2-face mode
  • Experimental active speaker detection (MAR + motion scoring)
  • Dead zone stabilization to prevent camera jitter

📝 Subtitle System

  • Rich preset system: Hormozi, MrBeast, Word Killer, TikTok, etc.
  • Live static HTML preview + slow animated video preview
  • Word-by-word highlight mode with dynamic resizing
  • Timeline-aware positioning for 2-face split layout
  • Async translation to 11 languages via deep-translator

⚡ GPU-Aware Encoding

  • Auto-detects NVIDIA NVENC, AMD AMF, Intel QSV, Apple VideoToolbox
  • Graceful CPU fallback (libx264 ultrafast)
  • Encoder result cached to avoid re-detection overhead
  • burn_subtitles.py only tries NVENC → CPU (misses AMD/Intel)

🔄 Smart Caching & Resumability

  • Skips re-download if input.mp4 already exists
  • Skips re-transcription if SRT/TSV/JSON already present
  • Prompts to reuse existing viral_segments.txt
  • Prompts to skip cutting if cuts/ folder is populated

🌐 WebUI — Gradio

  • Three workflows: Full, Cut Only, Subtitles Only
  • YouTube URL + Upload Video + Existing Project sources
  • Subtitle editor with Dataframe + per-segment re-render
  • Library tab with HTML video gallery (FastAPI static mount)
  • Process kill button + sticky auto-scroll log output

📦 Export & Integration

  • Premiere Pro XML export via export_xml_lib/
  • ZIP pack with video + ASS + JSON + face coords
  • FastAPI backend for static file serving
  • Modular CLI with 30+ argparse parameters

🛠️ Robustness Features

  • Raw segment repair — fixes AI output with missing timestamps
  • JSON recovery from truncated LLM responses (token limit)
  • DeepSeek <think> tag stripping from AI responses
  • Windows CP1252 emoji/encoding safe filename sanitization
  • yt-dlp with cookie fallback support

Issues & Improvement Opportunities — Full Catalog
# Category Severity File / Location Description & Recommendation
1 Security HIGH webui/app.py API key passed as CLI argument. --api-key <KEY> is visible in ps aux / Task Manager to any user on the machine. Fix: Pass via environment variable (env["GEMINI_API_KEY"] = api_key) and read it in main_improved.py using os.environ.
2 Security MEDIUM webui/app.py → convert_color_to_ass() Persistent debug log file in production. Every color conversion writes to debug_colors.log which grows indefinitely and may leak user color data to disk. Fix: Remove the with open("debug_colors.log") blocks entirely — the function is stable and doesn't need them.
3 Security MEDIUM api_config.json API keys stored in plain text JSON. File checked into the workspace root. Fix: Use a .env file (listed in .gitignore) and the python-dotenv library for key storage.
4 Security LOW scripts/transcribe_video.py Aggressive torch.load monkeypatch. Forces weights_only=False globally, bypassing PyTorch's security model safeguard against malicious pickle payloads in model files. Fix: Do this only for the specific affected loading calls, not globally.
5 Tech Debt MEDIUM webui/app.py → run_viral_cutter() --skip-prompts flag appended twice to the command list (lines ~253 and ~290). Harmless now but a quality signal. Fix: Remove the duplicate append.
6 Tech Debt MEDIUM main_improved.py Dead code: organize_output.organize() is commented out but the module is still imported. Fix: Either remove the import and comment, or re-enable the call.
7 Tech Debt MEDIUM main_improved.py Scattered print(f"DEBUG: ...") statements in production code paths (e.g. DEBUG: Checking input_video state, DEBUG: Download finished). Fix: Use Python's logging module with configurable levels (--verbose flag for debug output).
8 Tech Debt MEDIUM webui/app.py Global mutable current_process with no thread safety. If two browser tabs submitted jobs simultaneously, there would be a race condition on process management. Fix: Use threading.Lock or scope to a session-level object.
9 Tech Debt LOW scripts/burn_subtitles.py Burns only try NVENC → CPU libx264. The encoder detection in edit_video.py correctly handles AMD, Intel and Apple, but burn_subtitles.py skips them. Fix: Call the same get_best_encoder() function from edit_video.py instead of duplicating logic.
10 Tech Debt LOW webui/app.py temp_subtitle_config.json not cleaned up after processing. File accumulates in the workspace root. Fix: Delete the temp file in a finally block after the subprocess finishes.
11 Tech Debt LOW scripts/adjust_subtitles.py Subtitle mode fallback silently uses "no_highlight" for unrecognized mode strings (e.g. "word_by_word" mapped to "palavra_por_palavra" older alias, but not vice versa). Could silently produce wrong style. Fix: Normalize mode strings at ingestion time and log a warning for unknown values.
12 Tech Debt LOW main_improved.py Mixed language in code comments (Portuguese and English). Affects readability for international contributors. Comments should be standardized to English.
13 Tech Debt LOW scripts/cut_segments.py NVENC check is duplicated inside generate_segments() instead of using the shared get_best_encoder() from edit_video.py. Creates maintenance burden.
14 Performance HIGH main_improved.py Clips processed sequentially, not in parallel. For 10 clips, face crop and subtitle burn run one-by-one. Fix: Use concurrent.futures.ProcessPoolExecutor to process multiple clips simultaneously (limited to CPU core count).
15 Performance MEDIUM scripts/transcribe_video.py WhisperX model is not persisted between runs. Each processing job reloads the model from disk. For "Subtitles Only" re-runs, this is expensive. Fix: Cache the loaded model in a module-level variable and reuse it within the same process lifecycle.
16 Performance MEDIUM webui/library.py → generate_project_gallery() No caching of gallery HTML. The gallery re-scans the filesystem and re-parses JSON on every tab switch or refresh click. Fix: Add a simple mtime-based cache invalidation so the gallery is only rebuilt when files change.
17 UX MEDIUM webui/app.py No progress bar or ETA in the WebUI. Users see raw log text with no visual progress. Fix: Parse log lines for key milestones (Download %, Transcription %, Clip N/M) and feed a gr.Progress component or a dynamic HTML progress bar.
18 UX MEDIUM webui/app.py No confirmation on Stop button. A mis-click kills the running process permanently. Fix: Add a JS confirmation dialog or a two-step "Stop?" / "Confirm Stop" toggle.
19 UX LOW webui/app.py Segments field default is 3 with no max hint. New users don't know they can request 10, 20 or more. Fix: Add an info tooltip or placeholder text suggesting a typical range.
20 UX LOW webui/library.py Library tab doesn't display project metadata (total filesize, clip durations, creation timestamp, AI model used). Fix: Read the saved processing_config.json per project and render a summary card.
21 Feature MEDIUM All No batch/queue processing. Only one video can be processed at a time via the WebUI. Fix: Add a simple job queue (list of URLs/files) that processes sequentially, with status tracking per job.
22 Feature MEDIUM scripts/download_video.py Only YouTube is the tested download source. yt-dlp supports 1000+ sites (TikTok, Instagram, X/Twitter, etc.) but the UI only labels it "YouTube URL". Fix: Rename the field to "Video URL" and test/document other sources.
23 Feature MEDIUM main_improved.py No desktop notification when processing completes. Long jobs (30–60 min) require the user to actively check. Fix: Use Python's plyer library or a simple system tray notification when the subprocess exits.
24 Feature LOW scripts/edit_video.py Only 9:16 vertical output. No option for 1:1 (Instagram square) or 4:5 (portrait feed). Fix: Add an --output-ratio CLI argument and corresponding UI dropdown with presets.
25 Feature LOW webui/subtitle_editor.py No undo/redo in the subtitle editor. A wrong save overwrites the JSON permanently. Fix: Keep a backup copy on first load (*.json.bak) and add a "Restore Backup" button.
26 Feature LOW scripts/translate_json.py Translation uses only Google Translate (free tier) with no fallback. Rate limits can cause silent failures. Fix: Add a retry/backoff wrapper and expose an optional DeepL API key as alternative.
27 Feature LOW webui/app.py No watermark/branding overlay option. Content creators often need a channel watermark. Fix: Add an optional watermark image input and an FFmpeg overlay filter step in the burn pipeline.

Architecture Assessment

✅ What's Well Designed

  • Modular script separation: each pipeline stage is its own file with a clean entry function.
  • CLI + WebUI share the same main_improved.py entrypoint — no logic duplication.
  • i18n system integrated consistently across all layers (CLI, WebUI, scripts).
  • FastAPI static file serving is architecturally superior to Gradio's /file/ proxy for video playback.
  • Project folder structure is self-contained: input.mp4, cuts/, subs/, final/, burned_sub/ per project.

⚠️ Design Concerns

  • WebUI spawns main_improved.py as a subprocess — good isolation, but stdout streaming limits real-time progress granularity.
  • All shared state (subtitle config, temp files) written to the global WORKING_DIR, making multi-user deployment unsafe.
  • Color conversion logic duplicated between run_viral_cutter(), render_single() and render_all() in app.py — should be a shared helper.
  • burn_only_mode in main_improved.py is set from args.burn_only but burn_subtitles_option = True is hardcoded below — making burn_only redundant.
🏁
Executive Summary

ViralCutter is a legitimately impressive local-first AI video pipeline that goes far beyond a simple prototype. The core pipeline — download → transcribe → AI segment → cut → face-crop → subtitle → burn — is robust, well-tested, and handles a surprisingly wide range of edge cases (JSON recovery from truncated LLM output, face identity locking in split-screen, Windows encoding quirks, GPU encoder auto-detection).

Maturity assessment: Early Production / Advanced Beta. It is reliable enough for daily personal use and small community distribution, but has a handful of issues (especially the API key security and debug log leaks) that should be addressed before wider public release or any commercial deployment.

The #1 bottleneck for power users is sequential processing. Adding ProcessPoolExecutor to process multiple clips simultaneously is the single highest-ROI improvement — it would reduce total render time by 40–70% for typical 5–10 segment projects on multi-core machines.

The overall codebase is well-organized and maintainable, with a clear module structure and good separation of concerns. The scattered debug print() statements and duplicate logic are the main code quality concerns — none are dealbreakers, but cleaning them up would signal production readiness. The project has strong bones; the path to v1.0 is clear and achievable.
